# Request Channels

Request channels are request-scoped event streams between components (typically
agent -> dashboard), correlated by a unique ID generated at request time.

## Why not broadcast?

Broadcast patterns (e.g., `broadcastScoped()`) send events to all dashboard
clients and require clients to filter by session/wrapper ID. This has problems:

- No subscription lifecycle -- events fire whether anyone cares or not
- Race conditions when subscribing after the triggering action
- No cleanup signal -- clients guess when to stop listening
- Authorization is per-message, not per-channel

Request channels solve all of these.

## Pattern

```
Client                       Server                       Worker
   |                            |                            |
   |-- subscribe(channelId) --->|  (register WS)             |
   |                            |                            |
   |-- trigger(channelId) ----->|                             |
   |                            |-- forward(channelId) ----->|
   |                            |                            |
   |                            |<-- event(channelId) -------|
   |<-- event(channelId) -------|                            |
   |                            |                            |
   |                            |<-- result(channelId) ------|
   |<-- complete(channelId) ----|  (auto-detect or explicit) |
   |                            |                            |
   |-- unsubscribe(channelId) ->|  (cleanup)                 |
```

### Key properties

1. **Client generates the channel ID** -- subscribe before triggering to avoid
   race conditions
2. **Channel ID is the capability token** -- no additional auth needed on the
   event stream; auth happens at the trigger layer
3. **Server routes events only to subscribers** -- no broadcast waste
4. **Completion is server-detected** -- when the expected outcome occurs (e.g.,
   a session connects), the server sends a completion event
5. **Auto-cleanup** -- channels expire after a TTL, WS disconnect removes
   subscriber from all channels

## Current implementations

### Launch Jobs (`launch_log` / `job_complete` / `job_failed`)

Used for spawn/revive progress tracking.

**Channel ID:** `jobId` (UUID generated by dashboard)

**Subscribe:** `{ type: 'subscribe_job', jobId }`

**Trigger:** `POST /api/spawn { jobId }` or `{ type: 'revive_conversation', jobId }`

**Events:** Agent sends `{ type: 'launch_log', jobId, step, status, detail, t }`
at each pipeline stage (directory validation, spawn approval, tmux launch, etc.)

**Completion:** Concentrator detects session connect (meta handler matches
conversationId to jobId) and sends `{ type: 'job_complete', jobId, sessionId }`

**Failure:** Concentrator sends `{ type: 'job_failed', jobId, error }` on
spawn/revive failure

**Dashboard hook:** `useLaunchChannel(jobId)` -- subscribes/unsubscribes
automatically, returns `{ events, completed, failed, error, sessionId }`

## Adding a new request channel

1. Define the event types in `src/shared/protocol.ts`
2. Add channel storage to `src/broker/session-store.ts` (Map + cleanup)
3. Add subscribe/unsubscribe WS handlers
4. Add forwarding logic for worker events
5. Create a React hook in `web/src/hooks/` following `use-launch-channel.ts`
6. Wire the `use-websocket.ts` handler to dispatch events via CustomEvent
